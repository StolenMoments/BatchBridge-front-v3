# BatchBridge 화면 기획서

## 1. 문서 목적

이 문서는 BatchBridge 프런트엔드 개편을 위한 화면 기획서다. 목적은 현재 기능을 유지한 채, 제품을 "기본 CRUD 관리자 UI"에서 "반복 사용에 강한 운영 콘솔"로 재정의하는 것이다.

이 문서는 특정 프레임워크, UI 라이브러리, CSS 방법론에 종속되지 않는다. React, Vue, Svelte, 서버 렌더링 기반 템플릿, 또는 사내 디자인 시스템을 사용하더라도 본 문서의 화면 정의와 디자인 원칙을 그대로 구현할 수 있어야 한다.

## 2. 제품 정의

### 2.1 제품 성격

BatchBridge는 Claude, Grok, Gemini 등 usage-based AI API 사용자가 실시간 응답이 필요하지 않은 작업을 Batch API로 처리하고, 그 이력을 운영 관점에서 관리하는 도구다.

핵심 가치는 다음 네 가지다.

- 비용 절감: 실시간 호출보다 저렴한 배치 실행 경로 제공
- 운영 안정감: 제출, 진행, 완료, 실패를 한눈에 추적
- 반복 사용성: 하루에 여러 배치를 관리해도 피로하지 않은 UI
- 결과 검토 효율: 프롬프트별 입력과 결과를 빠르게 비교

### 2.2 주요 사용자

- AI API를 직접 사용하는 개발자, 운영자, 분석가
- 반복적인 비실시간 작업을 여러 건 생성하고 추적하는 사용자
- 비용, 처리량, 실패 재시도 여부를 함께 판단해야 하는 사용자

### 2.3 사용 맥락

- 단발성 탐색보다 반복적인 운영 업무 비중이 높다
- 여러 배치와 여러 프롬프트를 연속적으로 확인하는 경우가 많다
- 사용자는 "예쁜 화면"보다 "빠르게 읽히는 화면"을 더 선호한다

## 3. 디자인 철학

### 3.1 지향점

전체 경험은 **Refined Operational Console**을 지향한다. 화려한 SaaS 마케팅 톤이나 generic shadcn CRUD 톤이 아니라, 차분하고 촘촘하며 반복 사용에 적합한 도구형 인터페이스여야 한다.

### 3.2 핵심 원칙

1. 스캔 가능성이 장식보다 우선한다.
2. 상태와 액션은 미적으로 섞지 않고 구조적으로 분리한다.
3. 초록색은 브랜드의 중심축이되, 모든 강조를 독점하지 않는다.
4. 같은 앱 안의 리스트, 상세, 작성, 편집 화면은 하나의 레이아웃 언어를 공유해야 한다.
5. 각 화면은 "무엇을 읽고, 무엇을 결정하고, 무엇을 실행하는지"가 즉시 드러나야 한다.

### 3.3 금지 방향

- 카드 내부에 또 카드를 넣는 식의 중첩 컨테이너 구조
- 정보 밀도가 필요한 화면을 장식적 카드 그리드로 처리하는 방식
- 헤더, 본문, 상태, 액션이 모두 같은 강조도를 가지는 평면적 구성
- "기본 어드민"처럼 보이는 제네릭 타이포/배지/그리드 조합
- 모바일 대응을 이유로 핵심 기능을 숨기거나 축소하는 방식

## 4. 전역 UI 원칙

### 4.1 정보 구조 원칙

모든 주요 페이지는 아래 순서를 기본 구조로 삼는다.

1. 페이지 헤더
2. 현재 맥락을 조절하는 운영 툴바 또는 메타 스트립
3. 핵심 작업 영역
4. 보조 정보 또는 보조 액션

페이지 헤더는 항상 아래 요소를 가능한 한 일관되게 포함한다.

- 화면 제목
- 한 줄 설명
- 주요 액션 1개
- 보조 액션 또는 필터

### 4.2 레이아웃 언어

전역적으로 카드 사용 비중을 줄이고 아래 구조를 늘린다.

- dense list
- row-based list
- split pane
- meta strip
- sticky side panel
- section stack

카드는 "독립된 객체"를 강조할 때만 사용한다. 목록, 비교, 운영 히스토리처럼 스캔이 중요한 화면은 행 기반 구조를 우선한다.

### 4.3 타이포그래피 원칙

- 제목은 정보 구획을 분리하는 역할을 해야 한다
- 본문은 긴 문장보다 짧은 운영 문구와 메타 정보 중심으로 읽혀야 한다
- 배치명, 프롬프트명, 상태, 시간, 액션이 서로 다른 읽기 레벨을 가져야 한다
- 헤딩/브랜드용 서체와 본문용 서체를 분리해 위계를 명확히 한다

### 4.4 컬러 원칙

- 브랜드 green은 primary action, active state, 주요 링크, 선택 상태에 사용한다
- 상태색은 성공, 실패, 진행, 대기 의미를 전달하는 독립 축으로 운영한다
- 배경, 보더, 뮤트 톤에는 아주 약한 green tint를 적용해 전체 제품 정체성을 통일한다
- 상태색과 브랜드색이 충돌하지 않도록 역할을 분리한다

### 4.5 상호작용 원칙

- 사용자의 다음 행동이 예상 가능한 위치에 있어야 한다
- 비동기 작업에서는 진행 상태와 마지막 결과가 항상 드러나야 한다
- destructive action은 시각적으로 분리하되 과도하게 강조하지 않는다
- 펼침, 새로고침, 상태 변경은 즉각적인 피드백을 제공해야 한다

### 4.6 반응형 원칙

- 데스크톱에서는 운영 효율을 위해 2열 또는 split-pane 구조를 적극 사용한다
- 태블릿에서는 좌우 구조를 세로 스택으로 변환하되 정보 순서는 유지한다
- 모바일에서도 핵심 상태, 주요 액션, 현재 맥락은 접지 않는다

## 5. 공통 화면 프레임

### 5.1 앱 셸

앱 전체는 상단 글로벌 헤더, 본문, 하단 보조 푸터의 3층 구조를 가진다.

#### 글로벌 헤더 역할

- 브랜드 식별
- 1차 내비게이션
- 로케일 및 테마 제어
- 현재 도구가 "운영 콘솔"이라는 인상을 주는 안정적인 프레임 제공

#### 푸터 역할

- 기능성보다 제품 소속감과 마감 역할
- 과도한 정보 배치 금지

### 5.2 페이지 헤더 공통 규칙

각 페이지 헤더는 다음 질문에 답해야 한다.

- 지금 사용자가 어느 화면에 있는가
- 이 화면에서 무엇을 할 수 있는가
- 가장 중요한 1차 액션은 무엇인가

### 5.3 상태 표현 공통 규칙

Batch와 Prompt 모두 아래 상태 체계를 동일한 의미로 사용한다.

- Draft: 아직 제출 전이며 편집 가능
- In Progress: 외부 처리 진행 중
- Completed: 결과 수신 완료
- Failed: 실패가 확인된 상태
- Pending: Prompt 단위에서 아직 결과 미수신 상태

상태 표현은 색과 아이콘을 사용할 수 있으나, 핵심은 텍스트 레이블의 명확성이다.

## 6. 화면 정의

---

## 6.1 Batch List

### 목적

전체 배치를 빠르게 훑고, 현재 운영 상태를 파악하며, 다음으로 열어볼 배치를 즉시 선택하는 화면이다.

### 사용자의 핵심 질문

- 지금 어떤 배치들이 있는가
- 무엇이 진행 중인가
- 무엇이 실패했는가
- 어느 배치를 먼저 열어야 하는가

### 화면 구조

1. 페이지 헤더
2. 요약 메타 스트립
3. 필터/정렬/새로고침 툴바
4. 배치 목록 본문
5. 페이지네이션

### 필수 정보

- 배치명
- 모델
- 상태
- 프롬프트 수
- 생성 시각
- 성공/실패 개수
- 마지막 새로고침 또는 동기화 맥락

### 권장 레이아웃

- 목록 본문은 카드 그리드보다 row-based list 또는 table/list hybrid를 우선한다
- 한 행 안에서 좌측은 식별 정보, 중앙은 상태 및 수량, 우측은 시간 및 액션으로 정리한다
- 필터바는 상태 탭, 페이지 사이즈, 새로고침, 새 배치를 한 줄 툴바로 운영한다

### 주요 상태

- 초기 로딩
- 결과 없음
- 필터 결과 없음
- 일반 목록
- 네트워크 오류
- 삭제 진행 중

### UX 요구사항

- 목록은 한눈 비교가 가능해야 하며 세로 길이가 길어져도 피로하지 않아야 한다
- 진행 중 항목은 눈에 띄되 과하게 튀지 않아야 한다
- 실패 항목은 즉시 식별 가능해야 한다

## 6.2 Batch Create

### 목적

새 배치를 만들고 첫 번째 프롬프트를 함께 작성하는 화면이다. 작성 경험은 단순한 폼 입력이 아니라 "작업 제출 준비"에 가까워야 한다.

### 사용자의 핵심 질문

- 어떤 모델로 배치를 만들 것인가
- 어떤 프롬프트 유형을 보낼 것인가
- 지금 제출 가능한 상태인가

### 화면 구조

1. 페이지 헤더
2. Step 1. Batch setup
3. Step 2. First prompt
4. 보조 요약 패널 또는 sticky summary
5. 하단 sticky action bar

### 필수 정보

- 배치명
- 모델
- 첫 프롬프트명
- 시스템 프롬프트
- 사용자 프롬프트
- 프롬프트 타입
- 첨부/외부 컨텍스트
- edit 타입용 reference media

### 설계 원칙

- 작성 흐름은 두 단계로 인식되어야 한다
- 시스템 프롬프트는 optional이지만 발견 가능해야 한다
- 첨부와 외부 컨텍스트는 "보조 입력"으로 묶되 사용 부담을 낮춰야 한다
- 사용자는 스크롤 하단까지 가지 않아도 현재 입력 상태를 요약해서 볼 수 있어야 한다

### 주요 상태

- 모델 목록 로딩
- 기본 작성
- 텍스트 프롬프트 작성
- 이미지/비디오 edit 프롬프트 작성
- 첨부 업로드 대기
- 제출 중
- 검증 오류
- 생성 실패

### UX 요구사항

- primary action은 항상 현재 입력 가능 여부를 반영해야 한다
- 중요한 보조 정보는 collapse되더라도 존재를 알 수 있어야 한다
- 긴 폼이어도 현재 단계와 남은 판단 포인트가 명확해야 한다

## 6.3 Batch Detail

### 목적

하나의 배치를 중심으로 상태, 메타, 프롬프트 구성, 결과 검토, 재동기화 여부를 판단하는 핵심 운영 화면이다.

### 사용자의 핵심 질문

- 이 배치는 지금 어떤 상태인가
- 어떤 프롬프트들이 포함되어 있는가
- 어느 프롬프트가 성공/실패/대기 중인가
- 지금 내가 할 수 있는 다음 액션은 무엇인가

### 화면 구조

1. 상단 배치 메타 스트립
2. 주요 액션 영역
3. 배치 레벨 오류 또는 안내
4. 프롬프트 영역

상태에 따라 프롬프트 영역 구조는 달라진다.

#### Draft 상태

- 좌측: prompt list
- 우측: add prompt composer

#### Submitted 이후 상태

- stacked prompt sections
- prompt row header + expandable content 구조

### 필수 정보

- 배치명
- 상태
- 모델
- 생성, 제출, 완료 시각
- 배치 레벨 오류 메시지
- 프롬프트 목록 및 각 프롬프트 상태
- 제출, 동기화, 재동기화, 편집, 삭제 액션

### Draft 상태 설계 원칙

- 초안 단계에서는 "목록 관리"와 "추가 작성"을 병렬로 다룬다
- prompt list는 카드보다 리스트에 가깝게 구성한다
- 각 prompt는 번호, 이름, 타입, 첨부 여부, 짧은 preview를 우선 노출한다
- composer는 항상 작업 가능한 상태로 유지한다

### Completed/Failed 상태 설계 원칙

- 결과 검토는 카드 나열이 아니라 section stack 방식으로 읽혀야 한다
- 각 프롬프트는 접기 전에도 상태 판단이 가능해야 한다
- 입력, 첨부, 결과, 오류는 한 레벨 안에서 구분되어야 한다

### 주요 상태

- 초기 로딩
- 배치 없음
- Draft
- In Progress
- Completed
- Failed
- 일부 Prompt만 실패
- 재동기화 가능
- 저장/삭제/제출/동기화 진행 중

### UX 요구사항

- 배치 상태와 prompt 상태를 혼동하지 않게 분리한다
- 사용자가 상세 내용을 펼치지 않아도 전체 진행 상황을 추론할 수 있어야 한다
- destructive action은 Draft에서만 노출하거나 명확한 조건 하에서 노출한다

## 6.4 Prompt Detail

### 목적

단일 프롬프트의 입력과 출력, 메타 정보, 첨부 자료를 문서처럼 검토하는 화면이다.

### 사용자의 핵심 질문

- 이 프롬프트는 어떤 입력으로 실행되었는가
- 어떤 결과가 나왔는가
- 실패했다면 왜 실패했는가
- Draft 상태라면 수정 또는 삭제가 가능한가

### 화면 구조

1. 상단 헤더
2. 좌측 content viewer
3. 우측 meta sidebar

모바일에서는 content viewer가 먼저 오고 meta sidebar가 뒤따른다.

### 필수 정보

- 프롬프트명
- 상태
- 타입
- batch 연결 정보
- user prompt
- attachments
- model answer 또는 media result
- 오류 메시지
- 수정/삭제 액션 가능 여부

### 설계 원칙

- 입력과 출력은 같은 비중이 아니다
- 출력이 존재할 경우, 입력보다 결과를 더 쉽게 읽을 수 있어야 한다
- 문서 보기와 메타 보기의 역할을 분리한다
- markdown/text 전환은 시야를 흐리지 않는 segmented control 수준으로 정리한다

### 주요 상태

- 결과 대기
- 결과 완료
- 실패
- 미디어 결과
- Draft 편집 가능
- 존재하지 않음

### UX 요구사항

- 읽기 중심 화면이어야 하며 불필요한 컨테이너 분절을 줄인다
- 오류는 결과 영역 안에서 독립 블록으로 명확히 보여야 한다
- 편집 가능 조건은 상단에서 즉시 알 수 있어야 한다

---

## 6.5 Prompt Edit

### 목적

Draft 상태의 Prompt를 수정하는 화면이다. 작성과 비슷하지만, 더 높은 맥락성과 변경 인지성을 제공해야 한다.

### 사용자의 핵심 질문

- 지금 무엇을 수정 중인가
- 무엇이 바뀌었는가
- 저장 시 어떤 영향이 있는가

### 화면 구조

1. 페이지 헤더
2. 편집 폼 본문
3. 변경 요약 패널
4. 저장 액션 바

### 필수 정보

- 프롬프트명
- 소속 배치명
- 현재 상태
- 프롬프트 타입
- 시스템 프롬프트
- 사용자 프롬프트
- 첨부 및 외부 컨텍스트
- reference media
- 저장 가능 여부

### 설계 원칙

- create 화면의 단순 복제처럼 보이면 안 된다
- 상단에서 편집 대상과 상위 컨텍스트를 분명히 보여야 한다
- 변경된 첨부가 프롬프트 재생성으로 이어질 수 있다면 그 영향이 숨겨지면 안 된다
- 사용자는 저장 전에 "내가 어떤 변경을 만들고 있는지" 요약해서 볼 수 있어야 한다

### 주요 상태

- 초기 로딩
- 편집 가능
- 편집 불가
- 첨부 변경 감지
- 저장 중
- 저장 실패

### UX 요구사항

- 변경 요약은 실시간으로 업데이트되어야 한다
- 저장 버튼은 이 화면의 가장 명확한 primary action이어야 한다
- 편집 불가 조건은 사후 오류가 아니라 진입 시점에 명확히 알려야 한다

## 6.6 Templates

### 목적

재사용 가능한 Prompt 템플릿을 관리하고, 작성 시작점으로 활용하는 화면이다.

### 사용자의 핵심 질문

- 어떤 템플릿이 있는가
- 어떤 상황에서 써야 하는가
- 내용은 무엇인가
- 새 템플릿을 빠르게 만들 수 있는가

### 화면 구조

1. 페이지 헤더
2. 템플릿 목록
3. 선택 템플릿 미리보기
4. 생성/수정 편집 패널

### 권장 레이아웃

- 데스크톱: 좌측 목록 + 우측 preview split view
- 모바일: 목록 우선, 선택 시 preview 또는 inline expansion

### 필수 정보

- 템플릿명
- 설명
- 최근 수정일 또는 생성일
- system prompt 유무
- user prompt preview
- 생성/수정/삭제 액션

### 설계 원칙

- 템플릿은 단순 CRUD 데이터가 아니라 작업 출발점처럼 보여야 한다
- 목록의 각 항목은 선택 대상이어야 하며, 우측 preview가 내용을 읽어주는 구조가 적합하다
- 새 템플릿 작성은 필요 시 modal보다 side panel 또는 inline editor를 우선 검토한다

### 주요 상태

- 로딩
- 비어 있음
- 목록 존재
- 선택됨
- 생성 중
- 수정 중
- 삭제 중

### UX 요구사항

- 사용자는 편집 전에 전체 내용을 충분히 미리 볼 수 있어야 한다
- 수정/삭제보다 템플릿의 재사용 가치가 먼저 드러나야 한다

## 7. 공통 상태 및 메시지 원칙

### 7.1 Empty State

빈 상태는 "아무것도 없음"을 알리는 데서 멈추지 않는다. 각 화면의 첫 행동을 제안해야 한다.

- Batch List: 새 배치 생성 제안
- Batch Detail: 첫 Prompt 추가 유도
- Templates: 첫 템플릿 생성 또는 템플릿 활용 가치 설명

### 7.2 Error State

- 에러는 화면 전체를 붕괴시키기보다 해당 레벨에서 독립적으로 보여준다
- 배치 레벨 에러와 프롬프트 레벨 에러는 섞지 않는다
- 기술적인 메시지를 그대로 노출해야 할 때도, 사용자 판단에 필요한 문맥을 먼저 제공한다

### 7.3 Loading State

- 초기 로딩과 부분 로딩을 구분한다
- 새로고침, 저장, 삭제, 동기화는 전체 화면 블로킹보다 국소적 busy state를 우선한다

## 8. 컴포넌트 언어 가이드

구현 기술과 무관하게 다음 컴포넌트 역할군을 공통 언어로 삼는다.

- Page Header: 제목, 설명, 1차 액션을 담는 상단 영역
- Meta Strip: 상태, 모델, 시간, 개수 등 압축 메타 정보 영역
- Toolbar: 필터, 정렬, 새로고침, 페이지 크기 등 운영 제어 영역
- Row List: 비교와 스캔을 위한 목록 표현
- Section Stack: 상세 내용을 세로 흐름으로 읽는 구조
- Sticky Panel: 작성/요약/보조 조작을 고정하는 보조 패널
- Viewer Pane: 읽기 중심 콘텐츠 영역
- Summary Panel: 현재 선택/입력 상태를 요약하는 영역

## 9. 모션 및 피드백 원칙

- 모션은 장식보다 상태 전환 설명에 집중한다
- 새로고침, 아코디언 열림, 저장 성공, 동기화 진행은 명확한 피드백이 있어야 한다
- hover는 클릭 가능성과 행 선택 가능성만 전달하면 충분하다
- 과한 글로우, 과한 그라디언트, 과한 입체감은 지양한다

## 10. 접근성 및 국제화 원칙

- 한글과 영어 모두에서 제목, 버튼, 배지 폭이 무너지지 않아야 한다
- 상태 텍스트는 색만으로 구분하지 않는다
- 키보드만으로 주요 흐름 이동이 가능해야 한다
- sticky 영역과 split-pane 구조에서도 모바일 리더블리티를 유지해야 한다

## 11. 우선순위 제안

### 1차 개편

1. 전역 토큰 및 타이포그래피 재정의
2. 앱 셸과 페이지 헤더 패턴 통일
3. Batch List를 dense operational list로 재구성

### 2차 개편

1. Batch Detail 상단 메타 스트립 및 프롬프트 구조 재설계
2. Batch Create의 2-step 작성 구조와 요약 패널 도입
3. Prompt Edit를 맥락 중심 편집 화면으로 전환

### 3차 개편

1. Prompt Detail을 문서형 viewer로 개편
2. Templates를 split-view 자산 관리 화면으로 전환

## 12. 구현 시 준수사항

- API 스키마 변경 없이 프런트엔드 구조와 표현 방식을 바꾸는 것을 기본 전제로 한다
- 각 화면은 본 문서의 목적, 정보 우선순위, 상태 정의를 훼손하지 않는 범위에서 자유롭게 구현할 수 있다
- 실제 구현 단계에서는 이 문서를 기준으로 와이어프레임, 토큰 정의, 컴포넌트 추출 계획을 별도 작성한다

## 13. 기대 결과

개편이 완료되면 BatchBridge는 다음 인상을 주어야 한다.

- 여러 비동기 작업을 차분하게 관리할 수 있는 운영 도구
- 결과와 상태를 빠르게 읽을 수 있는 실무형 콘솔
- 기본 관리자 화면이 아니라, Batch API 사용 흐름에 맞게 설계된 제품
