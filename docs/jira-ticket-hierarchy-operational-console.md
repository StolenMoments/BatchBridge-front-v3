# BatchBridge 운영 콘솔 개편 Jira 상위/하위 구조

## 문서 목적

이 문서는 `jira-ticket-bodies-operational-console.md`에 정리한 실행 티켓들을 Jira에서 어떤 상위/하위 구조로 생성하면 좋은지 정리한 문서다. 실제 이슈키는 아직 없으므로, 제목 기준 계층과 권장 생성 순서를 함께 적는다.

## 권장 구조

- Epic 1개
- 상위 티켓 3개
- 실행 티켓 8개

이 구조를 권장하는 이유는 다음과 같다.

- 전체 목표를 한 Epic으로 묶을 수 있다
- 실제 스프린트 운영은 3개의 상위 묶음으로 관리할 수 있다
- 구현 티켓은 독립 검증 가능한 최소 단위로 유지할 수 있다

---

## 1. Epic

### Epic 제목

`[FE] BatchBridge 운영 콘솔 UX 개편`

### Epic 목적

- BatchBridge UI를 generic CRUD 관리자 화면에서 반복 사용에 강한 운영 콘솔로 전환
- 전역 디자인 시스템, 배치 운영 화면, 결과 검토/재사용 화면을 단계적으로 개편
- API 변경 없이 프런트엔드 구조와 표현 방식 개선

### Epic 하위 구성

- 상위 티켓 A. Foundation
- 상위 티켓 B. Batch Workflow
- 상위 티켓 C. Reading & Reuse

---

## 2. 상위 티켓 A. Foundation

### 상위 티켓 제목

`[FE] 운영 콘솔 전역 기반 정비`

### 목적

- 전역 토큰, 앱 셸, 페이지 헤더 패턴을 정비해 이후 화면 개편의 공통 기반 확보

### 하위 티켓

1. `[FE] 운영 콘솔 전역 토큰 및 테마 재정의`
2. `[FE] 운영 콘솔 공통 앱 셸 및 페이지 헤더 패턴 적용`

### 생성 순서

1. 상위 티켓 A 생성
2. 하위 티켓 1, 2 생성 후 A에 연결

### 완료 판단

- 전역 디자인 언어가 먼저 정리되어 이후 화면 티켓이 공통 기반 위에서 진행 가능

---

## 3. 상위 티켓 B. Batch Workflow

### 상위 티켓 제목

`[FE] 배치 운영 핵심 흐름 개편`

### 목적

- 배치 목록, 생성, 상세, 편집 등 반복 사용 빈도가 높은 핵심 운영 흐름 재구성

### 하위 티켓

3. `[FE] 배치 목록 화면 운영형 리스트 레이아웃 개편`
4. `[FE] 배치 상세 화면 상태판 및 작업면 구조 개편`
5. `[FE] 배치 생성 화면 2단계 작성 흐름 개편`
6. `[FE] 프롬프트 편집 화면 변경 요약 및 저장 흐름 개편`

### 권장 선행 관계

- 3, 4, 5, 6 모두 2번 티켓 이후 시작
- 6번 티켓은 5번 티켓과 같은 작성 패턴을 공유하므로 병렬 진행 가능
- 4번 티켓은 범위가 가장 크므로 별도 검수 포인트 설정 권장

### 생성 순서

1. 상위 티켓 B 생성
2. 하위 티켓 3, 4, 5, 6 생성 후 B에 연결

### 완료 판단

- BatchBridge 핵심 사용 흐름이 운영 콘솔 언어로 통일됨

---

## 4. 상위 티켓 C. Reading & Reuse

### 상위 티켓 제목

`[FE] 결과 검토 및 재사용 자산 화면 개편`

### 목적

- 결과 검토 경험과 템플릿 재사용 경험을 운영 콘솔 구조에 맞춰 고도화

### 하위 티켓

7. `[FE] 프롬프트 상세 화면 문서형 뷰어 레이아웃 개편`
8. `[FE] 템플릿 화면 split-view 자산 관리 구조 개편`

### 권장 선행 관계

- 7, 8 모두 2번 티켓 이후 시작 가능
- 7번은 4번의 상태 표현 방식과 시각적으로 유사해야 하므로 4번 완료 후 진행 권장
- 8번은 2번 이후 독립 진행 가능

### 생성 순서

1. 상위 티켓 C 생성
2. 하위 티켓 7, 8 생성 후 C에 연결

### 완료 판단

- 프롬프트 결과 검토와 템플릿 재사용이 단순 CRUD가 아니라 목적형 화면으로 정리됨

## 5. 전체 계층 구조표

```text
Epic
└─ [FE] BatchBridge 운영 콘솔 UX 개편
   ├─ [FE] 운영 콘솔 전역 기반 정비
   │  ├─ [FE] 운영 콘솔 전역 토큰 및 테마 재정의
   │  └─ [FE] 운영 콘솔 공통 앱 셸 및 페이지 헤더 패턴 적용
   ├─ [FE] 배치 운영 핵심 흐름 개편
   │  ├─ [FE] 배치 목록 화면 운영형 리스트 레이아웃 개편
   │  ├─ [FE] 배치 상세 화면 상태판 및 작업면 구조 개편
   │  ├─ [FE] 배치 생성 화면 2단계 작성 흐름 개편
   │  └─ [FE] 프롬프트 편집 화면 변경 요약 및 저장 흐름 개편
   └─ [FE] 결과 검토 및 재사용 자산 화면 개편
      ├─ [FE] 프롬프트 상세 화면 문서형 뷰어 레이아웃 개편
      └─ [FE] 템플릿 화면 split-view 자산 관리 구조 개편
```

## 6. Jira 등록 순서 추천

### 1단계

- Epic 생성
- 상위 티켓 A, B, C 생성

### 2단계

- 실행 티켓 1~8 생성
- 각 티켓을 상위 티켓에 연결

### 3단계

- 실행 티켓 선행 작업 필드를 실제 이슈키로 치환
- 필요 시 4번, 7번 티켓에 디자인 리뷰 subtasks 추가

---

## 7. 권장 매핑 규칙

### Epic 링크

- 1~8번 모든 실행 티켓은 동일 Epic에 연결

### Parent 링크

- 1, 2는 상위 티켓 A
- 3, 4, 5, 6은 상위 티켓 B
- 7, 8은 상위 티켓 C

### 선행 작업 치환 예시

- 2번 티켓 `선행 작업: DEV-XXX`
- 3번 티켓 `선행 작업: DEV-YYY`
- 4번 티켓 `선행 작업: DEV-YYY`
- 5번 티켓 `선행 작업: DEV-YYY`
- 6번 티켓 `선행 작업: DEV-YYY`
- 7번 티켓 `선행 작업: DEV-YYY`
- 8번 티켓 `선행 작업: DEV-YYY`

여기서 `DEV-XXX`는 1번 티켓, `DEV-YYY`는 2번 티켓의 실제 이슈키다.

---

## 8. 운영 메모

- 상위 티켓 본문은 간단히 목적, 범위, 포함 하위 티켓만 적는 것이 적절하다
- 실제 구현과 검증은 하위 실행 티켓에서 수행한다
- 4번 `배치 상세 화면 상태판 및 작업면 구조 개편`은 범위가 가장 넓으므로 스프린트 용량 기준으로 먼저 배정 여부를 확인하는 편이 안전하다
- 5번과 6번은 동일 담당자 또는 동일 스프린트로 묶는 것이 효율적이다
