## Design Context

### Users

BatchBridge는 Claude, Grok, Gemini 등 usage-based AI API를 사용하는 개발자, 운영자, 분석가를 위한 도구다. 이들은 실시간 응답이 필요하지 않은 작업을 Batch API로 처리하고, 여러 배치와 여러 프롬프트의 상태, 비용 효율, 처리량, 실패 여부를 운영 관점에서 연속적으로 확인한다. 핵심 사용 맥락은 탐색형 소비가 아니라 반복적인 운영 업무이며, 사용자는 하루에도 여러 번 배치 생성, 진행 상태 확인, 결과 검토, 실패 판단, 재시도 여부 결정을 수행한다.

### Brand Personality

브랜드 성격은 `정제된`, `운영 중심의`, `판독성 높은` 방향이다. 화면은 generic CRUD 관리자 UI나 과장된 마케팅형 SaaS가 아니라, 차분하고 촘촘하며 반복 사용에 피로가 적은 refined operational console이어야 한다. 사용자가 느껴야 할 감정은 신뢰, 통제감, 효율이다. “예쁜 화면”보다 “빠르게 읽히는 화면”이 우선이며, 시각적 개성은 정보 구조와 상태 표현의 정교함에서 나온다.

### Aesthetic Direction

전반 방향은 `Refined Operational Console`이다. 레이아웃은 카드 남용을 피하고 `meta strip`, `toolbar`, `row list`, `split pane`, `section stack`, `viewer pane` 같은 운영형 구조를 우선한다. Batch List는 dense row/table hybrid, Batch Detail은 상태에 따라 split pane 또는 section stack, Prompt Detail은 문서형 viewer, Templates는 split-view 자산 관리 화면으로 읽혀야 한다. 브랜드 그린은 제품의 정체성과 primary action, active state를 담당하지만, 성공/실패/진행 상태색과는 명확히 분리되어야 한다. light/dark 모두 지원하되 과한 glow, glassmorphism, 장식적 gradient, generic shadcn CRUD 인상은 피한다.

### Design Principles

1. 읽기 속도가 장식보다 우선이다. 사용자는 한 번에 많은 배치와 프롬프트를 비교하므로 정보 우선순위와 스캔 가능성이 최우선이다.
2. 상태와 액션은 구조적으로 분리한다. batch 상태와 prompt 상태, primary action과 destructive action, 요약 정보와 상세 내용이 시각적으로 섞이면 안 된다.
3. 카드보다 운영형 레이아웃을 우선한다. 목록, 비교, 검토, 작성은 각각 row list, split pane, section stack, summary panel 같은 목적형 구조로 설계한다.
4. 브랜드와 상태 색을 혼동하지 않는다. 브랜드 그린은 정체성과 선택 상태를, 상태 색은 성공/실패/진행 판단을 담당한다.
5. 반복 사용 피로를 줄인다. 빠른 판독, 예측 가능한 액션 위치, 밀도 있는 정보 배치, 로컬 busy state, 의미 있는 empty/error/loading 상태를 기본으로 한다.
